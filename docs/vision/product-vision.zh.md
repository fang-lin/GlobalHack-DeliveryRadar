# Delivery Radar — 产品愿景与范围（Vision & Scope）

> **权威: 中文（本文件） · 翻译: 英文（`product-vision.en.md`） · 最后更新: 2026-06-16 · 概念级、持续演化 · 两版冲突以中文为准**

## 0. 这份文档是什么

管理 Delivery Radar 的**概念级需求**——本质、定位、范围,而不是功能清单。它坐在需求规格（SRS, `docs/requirements/`）**之上**:概念在这里酝酿与打磨,**稳定后才蒸馏进 README / showcase**,定下来的关键决策抽成项目自己的 ADR。**不把还在演化的概念直接堆进 README。**

文档链(形状即 IIAC 本身):

> **Vision（本文,概念）→ ADR（定下来的决策）→ 需求规格（怎么造）→ README / showcase（对外蒸馏）**

## 1. 一句话愿景

Delivery Radar 是**面向 AI 时代软件生产的「过程控制」(process control)**:把团队**记录的意图**当作规格,让每一次变更——无论出自人还是 agent——都朝它**收敛**。

## 2. 问题（Why）

AI 让代码以机器速度产出,人类已无法在同样速度上"对照当初的为什么"做评审;意图散落在 ADR、人脑、会议室里,实现于是**悄悄漂移**。现有手段都不看"还符不符合当初的决定":测试看"能不能跑",linter 看"整不整洁",通用 AI review 给"看似合理的意见"(甚至会主动建议一个违反决定的改法)。

## 3. 本质:一套过程控制工艺（核心 reframe）

它**不是"又一个 review 工具"**,而是一套**过程控制工艺 / 过程纪律**:定规格 → 控制过程使产出合规格 → 量偏差 → 纠偏 → 收敛到规格。映射:

| 制造业的工艺 / 过程控制 | Delivery Radar（IIAC） |
|---|---|
| 规格(spec) | 记录的意图(ADR + driver) |
| 生产过程 | 开发(人 / agent 写码) |
| 质检 | 对照意图的 conformance 检查 |
| 偏差 | drift |
| 纠偏 | remediation / supersede（过人确认门） |
| 收敛到规格 | 代码库随时间朝意图收敛 |

slide 上那句 *"convergence makes the trajectory settle — no oscillation"* 本就是**控制论 / 过程控制(SPC)**的语言——这是埋好的,不是新加的比喻。
（措辞:对外用 **"process control / engineering discipline for intent-aligned software production"**;中文偏 **"过程控制工艺 / 过程纪律"**,少用单字"工艺"以免"手工匠人"的味道,与"机读、机器可检"的调性冲突。）

## 4. 两个硬核 + 收敛闭环（牙齿——否则就成空谈）

抽象的定位只有落在这几样硬核上才不是 buzzword:

- **硬核 ①**:意图**机读 + 绑定业务 driver**(ADR 里的 constraint 块,稳定 ID)。
- **硬核 ②**:**对照在 scope 内的意图、并带 driver 理由**做语义判断——**不是对照最佳实践**。"对齐意图 ≠ 对齐最佳实践"是灵魂句。
- **收敛闭环**:每过一次对齐一个改动 → 时间上整体收敛;任何**写回意图**都过**人确认门**;全程**留痕可审计**(收敛需要记忆)。

## 5. 形态只是外在体现（CLI / Skill / Agent）

同一套工艺,插在软件生命周期的不同触点上;**不绑死任何形态 → 更耐久**(工具会换,工艺留下):

- **CLI**:接在 CI 上的关卡（今天的实现)。
- **Skill**:让 dev / agent **装上**这套纪律(方法 + 驱动 radar)。
- **Agent**:自主**自检**——编码 agent 在开 PR 前用同一套判断逻辑检查自己。

由此,**radar = "agent 化开发"循环里的对齐层 / agent 的对齐自检**。

## 6. 双路径 + 可审计（北极星）

- **人类今天**:agent 写码、人类实时 steering——不可规模化(一人一会话)。
- **探索目标(非承诺)**:**长程自治**——记录的、机器可检的意图替代实时 steering,agent 开 PR 前自检、只在决策点上报人类。**对齐由基础设施承担,人退出循环、对齐留在循环里。**
- **全程可审计**:谁决定 · 改了什么 · 为什么——把"某一刻对齐"变成"一条收敛轨迹"。

## 7. 范围（是什么 / 不是什么）

**是**:意图对齐的过程控制;**advisory(不拦合并)**;证据化(裁定带 ADR 条款 + 代码行 + 修复方向)。

**不是**:不是 linter / 测试的替代;不是通用 AI code review;**默认不做强制 gating**——gating 要"既确定性、又在本仓历史上验证过精度"才**凭实绩赚得**。

## 8. 概念级需求条目（tracked）

以稳定 ID 管理,便于后续细化 / 取舍:

| ID | 概念级主张 | 状态 |
|---|---|---|
| **VIS-1** | 本质是**过程控制工艺**,不是工具 | 采纳 |
| **VIS-2** | **形态(CLI / Skill / Agent)是外在体现**,不绑死;同一套工艺多触点 | 采纳 |
| **VIS-3** | **radar = agent 化开发的对齐层 / agent 的对齐自检** | 采纳 |
| **VIS-4** | **Dogfood**:用项目自己的 ADR(+ constraint 块)记录并**让 radar 检查项目自身**是否偏离自己的意图 | 探索 |

## 9. 诚实边界

- "过程控制 / 工艺 / 方法论"这类词容易听着空——**只有每次都落到硬核 ①②(机读意图 + driver 锚定)+ 收敛闭环上**,才不是 buzzword。
- demo / 证据必须继续落到具体(radar 抓住一个真实违规;评测台量化 grounded vs ungrounded),否则一抽象就变哲学。
- 长程自治是**探索方向**,不是已交付能力;对外措辞不得把它说成"已实现"。
